# Frequently Asked Questions

This document addresses common questions from contributors to BuilderHub.

## General Questions

### What is BuilderHub?

BuilderHub is a community-driven learning platform that provides a structured roadmap to help beginners go from zero coding confidence to contributing to real fullstack applications. We focus on practical, hands-on learning through project contributions.

### Who can contribute?

Anyone! We welcome contributors of all skill levels. Our learning roadmap is designed to take complete beginners and guide them to making meaningful contributions.

### Do I need to be an expert to contribute?

Absolutely not! We have contributions for all skill levels:

- **Beginners**: Fix typos, improve documentation, add examples
- **Intermediate**: Implement features, write tests, review code
- **Advanced**: Architecture decisions, complex features, mentoring

## Contributing

### How do I find something to work on?

1. Check [open issues](https://github.com/kc-clintone/builderhub/issues) labeled "good first issue"
2. Look for issues labeled "help wanted"
3. Check the [learning roadmap](roadmap.md) for phase-appropriate tasks
4. Join [discussions](https://github.com/kc-clintone/builderhub/discussions) to ask for guidance

### What makes a good contribution?

- **Clear purpose**: Addresses a real need or problem
- **Well documented**: Includes necessary documentation
- **Tested**: Works as expected
- **Follows guidelines**: Matches our coding and contribution standards
- **Helpful**: Benefits the community

### How do I write a good commit message?

Follow conventional commit format:

```
type(scope): description

[optional body]

[optional footer]
```

Examples:

- `feat: add user authentication system`
- `fix: resolve typo in getting started guide`
- `docs: update contribution guidelines`

Types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`

### What if I'm not sure about my contribution?

- Create a draft pull request and ask for feedback
- Join our discussions to get input
- Start small with documentation or typo fixes
- Ask questions â€“ we're here to help!

## Technical Questions

### What technologies does BuilderHub use?

Currently, BuilderHub is primarily documentation-focused. Future development may include:

- **Frontend**: HTML, CSS, JavaScript, React
- **Backend**: Node.js, Express, databases
- **Tools**: Git, GitHub, VS Code

### How do I test my changes?

For documentation changes:

- Check links work
- Verify formatting
- Read through for clarity

For code changes (when applicable):

- Run existing tests
- Add new tests if needed
- Manual testing

### I found a bug. What should I do?

1. Check if it's already reported in [issues](https://github.com/kc-clintone/builderhub/issues)
2. If not, create a new issue with:
   - Clear description
   - Steps to reproduce
   - Expected vs actual behavior
   - Environment details

## Pull Requests

### How long does PR review take?

We aim to review PRs within 1-2 weeks, but it depends on reviewer availability. Feel free to ping for updates after a week.

### What happens if my PR has conflicts?

1. Update your branch: `git pull origin main`
2. Resolve conflicts in your editor
3. Test your changes
4. Push the updated branch

### Can I update my PR after creating it?

Yes! Push additional commits to your branch. The PR will update automatically. Use "force push" only if absolutely necessary and you've communicated it.

## Advanced Topics

### How do I become a maintainer?

Maintainers are contributors who show:

- Consistent quality contributions
- Helpful community involvement
- Understanding of project goals
- Responsible PR reviews

We don't have formal requirements â€“ it's about demonstrated commitment.

### Can I propose major changes?

For significant changes:

1. Start a discussion first
2. Create an issue with detailed proposal
3. Get community feedback
4. Implement incrementally

### How does decision making work?

- **Documentation**: Open to community contributions
- **Features**: Discussed in issues/discussions
- **Architecture**: Consensus among active maintainers
- **Policy**: Community discussion and maintainer approval
