# Maintainers Guide

This document outlines the responsibilities and processes for BuilderHub maintainers.

## Role Overview

Maintainers are trusted contributors who have demonstrated:

- Consistent quality contributions
- Understanding of project goals and vision
- Responsible code review and community interaction
- Commitment to the project's values

## Responsibilities

### Code Review

- Review pull requests within 1-2 weeks
- Provide constructive, actionable feedback
- Ensure code quality and adherence to standards
- Test changes when possible
- Approve and merge PRs that meet requirements

### Issue Management

- Triage new issues and assign appropriate labels
- Respond to questions and provide guidance
- Close duplicate or invalid issues
- Escalate security issues appropriately

### Community Management

- Moderate discussions and maintain code of conduct
- Help new contributors get started
- Resolve conflicts constructively
- Foster inclusive community environment

### Project Management

- Participate in decision-making processes
- Help plan releases and milestones
- Update documentation as needed
- Monitor project health and metrics

## Processes

### Pull Request Review Process

1. **Initial Review** (1-2 days)
   - Check PR description and requirements
   - Review code changes for correctness
   - Run automated tests if applicable

2. **Detailed Review** (2-5 days)
   - Code quality and style check
   - Documentation review
   - Testing verification
   - Security considerations

3. **Approval/Merge**
   - Approve if all criteria met
   - Merge using appropriate method
   - Close related issues

### Issue Triage Process

1. **Label Assignment**
   - `bug`: Actual bugs in functionality
   - `enhancement`: New features or improvements
   - `documentation`: Documentation issues
   - `question`: Questions or support requests
   - `good first issue`: Suitable for newcomers

2. **Priority Assessment**
   - `priority-high`: Critical issues
   - `priority-medium`: Important but not urgent
   - `priority-low`: Nice to have

3. **Assignment**
   - Assign to appropriate maintainer or contributor
   - Set milestones for tracking

### Release Process

1. **Planning** (1-2 weeks before)
   - Review changelog and pending PRs
   - Coordinate with contributors
   - Plan release notes

2. **Preparation**
   - Update version numbers
   - Final testing
   - Documentation updates

3. **Release**
   - Create and push git tag
   - Publish GitHub release
   - Announce in discussions

## Communication Guidelines

### Response Times

- Issues: Acknowledge within 48 hours
- PRs: Initial review within 1 week
- Security issues: Immediate response
- General questions: Within 24 hours

### Communication Style

- Be respectful and professional
- Provide clear, actionable feedback
- Assume good intent
- Use inclusive language
- Be patient with newcomers

## Tools and Access

### Required Access

- Write access to main repository
- Access to GitHub organization (if applicable)
- Access to CI/CD systems
- Access to documentation platforms

### Recommended Tools

- GitHub Desktop or command line Git
- Code editor with Git integration
- Markdown preview tools
- Project management tools

## Onboarding New Maintainers

1. **Nomination**: Current maintainer identifies potential maintainer
2. **Discussion**: Team discusses candidate's contributions and fit
3. **Trial Period**: 1-3 months with mentorship
4. **Formal Approval**: Majority vote of current maintainers
5. **Handover**: Transfer knowledge and access

## Offboarding

If a maintainer needs to step down:

1. **Announcement**: Public announcement in discussions
2. **Knowledge Transfer**: Document processes and hand off responsibilities
3. **Access Removal**: Remove repository access
4. **Recognition**: Thank them for their service
